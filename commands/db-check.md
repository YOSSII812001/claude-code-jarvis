# データベースとマイグレーションの整合性確認

## ゴール

データベースカラム追加後に、ローカルおよびリモートデータベースのスキーマとマイグレーションファイルの整合性を確認し、不整合があれば修正する。

## 確認項目

1. **マイグレーションファイルの確認**
   - 最新のマイグレーションファイルが存在するか
   - マイグレーションファイルに追加カラムの定義が含まれているか
   - マイグレーションの命名規則は適切か

2. **ローカルデータベース確認**
   - ローカルDBに追加カラムが存在するか
   - カラムの型、制約が正しいか
   - インデックスが適切に設定されているか

3. **リモート（Supabase）データベース確認**
   - リモートDBに追加カラムが存在するか
   - ローカルとリモートのスキーマが一致しているか
   - RLSポリシーが適切に設定されているか

## 実行ステップ

### Phase 1: マイグレーションファイルの確認

1. `docs/migrations/` または該当するマイグレーションディレクトリを確認
2. 最新のマイグレーションファイルを読み取る
3. 追加されたカラムの定義を確認
4. SQLの構文エラーがないか確認

### Phase 2: ローカルデータベースの確認

1. データベース接続情報を確認（.env ファイル）
2. 対象テーブルのスキーマを取得
   ```sql
   SELECT column_name, data_type, is_nullable, column_default
   FROM information_schema.columns
   WHERE table_name = '対象テーブル名'
   ORDER BY ordinal_position;
   ```
3. 追加カラムが存在するか確認
4. カラムの属性がマイグレーションファイルと一致しているか確認

### Phase 3: リモートデータベースの確認

1. Supabase接続情報を確認
2. リモートDBのスキーマを取得（Phase 2と同様のクエリ）
3. ローカルとリモートのスキーマを比較
4. 差分があれば報告

### Phase 4: 不整合の修正

1. **マイグレーションファイルに問題がある場合**
   - 修正されたマイグレーションファイルを作成
   - バックアップを推奨

2. **ローカルDBに不整合がある場合**
   - マイグレーションの再実行を提案
   - または手動でのALTER TABLE文を提供

3. **リモートDBに不整合がある場合**
   - Supabaseダッシュボードでの手動修正を提案
   - またはマイグレーションSQLの実行手順を提供

## 出力形式

### 整合性確認レポート

```
=== データベース整合性確認レポート ===

【マイグレーションファイル】
✅ ファイルパス: docs/migrations/YYYYMMDD_add_columns.sql
✅ 追加カラム: column1, column2, column3

【ローカルデータベース】
✅ テーブル: table_name
✅ カラム存在確認: OK
⚠️ カラム型不一致: column2 (expected: VARCHAR(255), actual: TEXT)

【リモートデータベース (Supabase)】
❌ カラム未追加: column3が存在しません
❌ RLSポリシー未設定

【推奨アクション】
1. ローカル: column2の型をVARCHAR(255)に変更
   ALTER TABLE table_name ALTER COLUMN column2 TYPE VARCHAR(255);

2. リモート: 以下のマイグレーションを実行
   - Supabaseダッシュボード > SQL Editor で実行
   - または supabase migration apply コマンド実行

3. RLSポリシー設定が必要です
```

## 注意事項

- データベースへの変更は慎重に行うこと
- 本番環境への適用前に必ずバックアップを取得すること
- RLS（Row Level Security）ポリシーの確認を忘れないこと
- インデックスのパフォーマンス影響を考慮すること

## 使用例

```
/db-check
```

または、特定のテーブルを指定する場合:

```
/db-check companies
/db-check csf_proposals
```
