# GitHub Issue実装フロー: $ARGUMENTS

## ゴール

指定されたIssueを**計画→サブエージェントレビュー→実装**の自動フローで実施する。

## 手順

Issue番号: $ARGUMENTS

以下のフローを**自動的に**実行する。ユーザーへの確認はステップ④の承認時のみ。

### ⓪ スキルの同時起動

フロー開始時に、以下の3つのスキルを**Skillツールで同時に（並列で）**起動する：

1. **usacon** — プロジェクト固有のルール・構成・トラブルシューティングを参照
2. **codex-autopilot** — Codex CLIを自動運転モードで活用し、ユーザー不在でも開発を継続
3. **agent-teams** — Agent Teamsによる並列実装で、ファイル競合を防ぎつつ効率的に分担

3つのSkillツール呼び出しは依存関係がないため、**1つのメッセージで3つ同時に呼び出す**こと。

### ① Issue詳細を取得

```bash
gh issue view $ARGUMENTS --json title,body,labels,assignees,milestone
gh pr list --state open
```

**画像がある場合**: Issue本文に `user-attachments/assets` のURLが含まれていたら、
`github-cli` スキルの「Issue内の画像取得」手順に従って画像をダウンロード・確認すること。
スタイル崩れやUI修正のIssueでは画像確認が必須。

### ② コードベース調査 → 実装計画の作成

Glob、Grep、Readツールを使ってIssue要件に関連するコードを徹底調査し、以下の構成で計画を作成する：

- **Issue概要**: 要件の要約
- **影響範囲分析**: 変更対象ファイル、影響を受ける既存機能、DB変更の有無
- **実装方針**: アプローチと設計理由
- **実装ステップ**: 具体的な作業の順序
- **リスク・注意点**: エッジケース、破壊的変更の有無

### ③ サブエージェントでプランレビュー（軽量・差分ベース）

Taskツール（subagent_type: general-purpose）でレビューを起動。

**重要なルール（トークン節約）:**
- 計画内容と関連コードの抜粋を**プロンプトに直接埋め込む**こと
- サブエージェントに「ファイルを読んでレビューして」と指示してはいけない
- サブエージェントがコードベースを探索する必要がないようにデータを渡す

**レビュー観点（4つに絞る）:**
- アプローチはIssue要件を満たしているか
- 既存アーキテクチャと整合しているか
- 変更対象ファイルに漏れはないか
- DB変更時: マイグレーション、RLS、型生成の確認

評価は ✅問題なし / ⚠️要検討 / ❌要修正 の3段階。

### ④ レビュー結果の反映 → ユーザー承認

❌の指摘があれば計画を修正した上で、レビュー結果と最終計画をユーザーに提示。
承認を得てからステップ⑤に進む。

### ⑤ 実装

承認後、計画に沿って実装を開始する。
usaconプロジェクトの場合は、staging経由フロー・クアドレビュー（Lint + /sub-review + Codex差分レビュー + CodeRabbit）・changelog更新に従う。
