## 構造化コードレビュー（5エージェント並列 + 信頼度スコアリング）

> Anthropic公式 `/code-review` プラグインのアーキテクチャを基に、差分埋め込み方式で最適化。
> 4つの専門Claudeエージェント + 1つのCodexエージェントが並列にレビューし、信頼度スコアで誤検知をフィルタリングする。
> 異なるLLM（Claude + Codex）のクロスバリデーションにより、単一モデルでは見逃しやすい問題を検出する。

以下の手順でレビューを実行すること：

### Step 1: 差分情報の取得

Bashで以下を実行し、変更内容を把握する：

```bash
# 変更ファイル一覧
git diff --stat HEAD~1

# 変更内容の詳細
git diff HEAD~1
```

- 引数に `--base <branch>` が指定されている場合は `HEAD~1` の代わりにそのブランチとの差分を使用する
  - 例: `/sub-review --base staging` → `git diff staging...HEAD`
- 引数に `--no-codex` が指定されている場合は Agent 5（Codex）をスキップし、Agent 1〜4 のみで実行する
  - 例: `/sub-review --no-codex`（autopilot-issue 経由など、外部で別途 Codex を起動する場合に使用）
- 変更ファイルが多い場合（10ファイル以上）は重要なファイルに絞る
- diffが500行を超える場合はファイルごとに分割してレビューする

### Step 1.5: diffサイズゲート（コスト最適化）

Step 1で取得した diff の **変更行数** に基づいてレビュー戦略を自動調整する:

| 変更行数 | 戦略 | エージェント数 | 理由 |
|----------|------|---------------|------|
| **0行**（差分なし） | **即終了** | 0 | レビュー対象がない |
| **1〜10行** | **簡略レビュー** | 2（Agent 2: バグ + Agent 4: 規約） | 小規模変更にフルレビューは過剰。バグ検出と規約準拠の2観点で十分 |
| **11行以上** | **フルレビュー** | 5（通常通り） | 標準フロー |

- 変更行数は `git diff --stat` の最終行（`X files changed, Y insertions(+), Z deletions(-)`）から `Y + Z` で算出
- `--full` オプション指定時はサイズに関わらずフルレビューを強制
- 簡略レビュー時は Step 5 の統計にも `(simplified: ≤10 lines)` と付記する

### Step 2: CLAUDE.md / プロジェクト規約の確認

変更ファイルが存在するディレクトリおよびルートの CLAUDE.md を Read ツールで確認する。
Agent 4（規約準拠チェック）のプロンプトに含めるため、規約の要点を把握しておく。

### Step 3: エージェントを並列起動

以下のエージェントを **1つのメッセージで並列起動** する。
- Agent 1〜4: Agent ツール（subagent_type: general-purpose）— Claude による専門レビュー
- Agent 5: Agent ツール（subagent_type: general-purpose）— Codex CLI による差分レビュー

**`--no-codex` が指定されている場合は Agent 5 をスキップし、Agent 1〜4 のみを起動する。**

Agent 1〜4 のプロンプトに git diff の結果と変更ファイル一覧を直接埋め込むこと（Agent 5 は Codex 自身が diff を取得するため埋め込み不要）。

**Agent 1〜4 共通の指示（各プロンプトの冒頭に含めること）:**

```
必要な情報はすべてこのプロンプト内に含まれています。
コードベースの探索は不要です。Glob/Grep/Read ツールは使用しないでください。

【信頼度スコア基準（0-100）】
- 0-25: 推測レベル（根拠が弱い、コンテキスト不足で判断不可）
- 26-50: 可能性あり（要追加調査、確証なし）
- 51-75: 高確度（明確な根拠あり、だが例外の可能性）
- 76-100: 確実（コードから直接証明可能、反例なし）

【出力形式】
各指摘に対して:
- ファイル名: 該当ファイル
- 該当箇所: 変更行の説明
- 問題: 問題の内容（1-2文）
- 信頼度: スコア（0-100）と根拠（1文）
- 修正案: 信頼度75以上の場合のみ、具体的なコード例

問題がない場合は「重大な問題なし」と1行で報告。

【誤検知フィルタ（以下は指摘しない）】
- PR以前から存在する問題（差分で導入されたものだけを対象）
- バグに見えるが実際には正しいパターン
- リンターが検出する問題（リンターに任せる）
- 主観的なコードスタイルの好み
- lint ignoreコメントで明示的に抑制されている問題
- 一般的なベストプラクティス違反（CLAUDE.mdに明記されていない限り）

【プロジェクト固有の高信号基準（必ずフラグ）】
- バックエンドAPIで `supabaseAdmin` の代わりに `supabase`（Anon Key）を使用している
- テーブルクエリで `.eq('org_id', orgId)` フィルタが欠落している（テナント間データ漏洩）
- Stripe Webhook署名検証（`stripe.webhooks.constructEvent()`）のバイパスや欠落
- `SUPABASE_SERVICE_ROLE_KEY` / `STRIPE_SECRET_KEY` / `ANTHROPIC_API_KEY` がフロントエンドコードや `VITE_` プレフィックス変数に露出
- `getUserIdFromRequest()` を経由せず `req.headers['x-user-id']` を直接使用、または `DEFAULT_USER_ID` フォールバックの本番混入
- `billing` スキーマテーブルへのRLSポリシー変更（service_role限定アクセスの意図的設計を崩す変更）
- `safeRedirectToStripe()` のURLバリデーション迂回
- クレジット消費（`creditService.consumeCredit()`）のトランザクション整合性欠如（失敗時のロールバック漏れ）

【プロジェクト固有の除外ルール（誤検知として無視）】
- `supabaseAdmin`（Service Role Key）をバックエンド全体で使用 → 意図的な設計
- `.schema('billing')` によるクロススキーマクエリ → billing スキーマの正常パターン
- `billing_*_view` が `service_role` のみに SELECT 権限 → フロントエンド直接参照防止の設計
- SSEレスポンスで `res.write()` 後に明示的 `res.end()` がない → `streamCompleted` フラグと `req.on('close')` で管理
- `isConnectionClosed` フラグによるSSE中断パターン → 正常な接続クリーンアップ
- フロントエンドの `supabase` クライアントが `null` → `isSupabaseEnabled` が `false` の開発環境パターン
- `DEFAULT_USER_ID` / `DEFAULT_ORG_ID` のハードコード → ローカル開発用モック認証（本番ではJWT強制）
- `localStorage.getItem('token')` の毎回読み取り → Supabaseセッション + APIサーバーJWTの二重トークン設計
```

#### Agent 5: Codex差分レビュー（Cross-LLM Validation）

**`--no-codex` 指定時はこのエージェントをスキップする。**

Codexの出力はメインコンテキストを圧迫するため、**必ず Agent ツール（subagent_type: general-purpose）経由**で実行する。
サブエージェント内で Bash ツールを使い `codex exec` を呼び出す。

**diff渡し方式**: コマンドライン引数にdiffを含めると Windows 8191文字制限に抵触するため、
Codex自身にプロジェクトディレクトリ内で `git diff` を実行させる。

```
Task(
  subagent_type="general-purpose",
  description="Codex差分レビュー",
  prompt="""
  以下のBashコマンドを実行し、結果を信頼度スコア付きで要約してください。

  ## 実行コマンド
  ```bash
  codex exec --full-auto --sandbox read-only \
    --cd "{プロジェクトディレクトリ}" \
    "git diff {diffベース} を実行し、その差分をレビューしてください。
     バグ、セキュリティ問題、パフォーマンス問題、設計上の懸念を指摘し、
     各指摘に信頼度スコア（0-100）を付けてください。
     信頼度基準: 0-25=推測, 26-50=可能性あり, 51-75=高確度, 76-100=確実。

     【必ずフラグすべき高信号パターン】
     - バックエンドAPIで supabaseAdmin の代わりに supabase（Anon Key）を使用
     - テーブルクエリで .eq('org_id', orgId) フィルタが欠落（テナント間データ漏洩）
     - Stripe Webhook署名検証のバイパスや欠落
     - シークレットキーがフロントエンドコードや VITE_ プレフィックス変数に露出
     - getUserIdFromRequest() を経由せず req.headers['x-user-id'] を直接使用

     【誤検知として無視すべきパターン】
     - supabaseAdmin（Service Role Key）をバックエンド全体で使用 → 意図的な設計
     - DEFAULT_USER_ID / DEFAULT_ORG_ID のハードコード → ローカル開発用
     - localStorage.getItem('token') の毎回読み取り → 二重トークン設計

     問題がなければ「重大な問題なし」と報告してください。
     確認や質問は不要です。具体的な提案・修正案・コード例まで自主的に出力してください。"
  ```

  タイムアウト: 300秒

  ## 出力形式
  Codexの結果を以下の形式に変換して報告:
  - ファイル名: 該当ファイル
  - 該当箇所: 変更行の説明
  - 問題: 問題の内容（1-2文）
  - 信頼度: スコア（0-100）と根拠（1文）
  - 修正案: 信頼度75以上の場合のみ、具体的なコード例

  問題がない場合は「重大な問題なし」と1行で報告。
  """
)
```

**注意:**
- `{diffベース}` には Step 1 で決定した値を使用する（デフォルト: `HEAD~1`、`--base` 指定時: `staging...HEAD` 等）
- `{プロジェクトディレクトリ}` は `git rev-parse --show-toplevel` で取得する
- Codex自身が `git diff` を実行するため、diff をコマンドライン引数に含めない（Windows 8191文字制限回避）
- Codexの出力が長大でもサブエージェントが要約するためメインコンテキストは保護される

### Step 4: 結果の統合とフィルタリング

5つのエージェント（Agent 1〜4: Claude, Agent 5: Codex）の結果を受け取った後：

1. **信頼度80未満の指摘を除外する**
2. 複数エージェントから同じ指摘がある場合は統合する（信頼度は最高値を採用）
3. Claude（Agent 1〜4）とCodex（Agent 5）の両方から同じ指摘がある場合は **クロスバリデーション済み** とマークする
4. 残った指摘を信頼度の降順でソートする

### Step 5: レビュー結果の報告

以下の形式でユーザーに報告する：

**指摘がある場合:**
```
## 🔍 構造化コードレビュー結果

### 検出された問題（信頼度≥80）

| # | カテゴリ | ファイル | 問題 | 信頼度 | クロス検証 |
|---|---------|---------|------|--------|-----------|
| 1 | バグ    | src/... | ...  | 95     | ✅ Claude+Codex |
| 2 | 設計    | src/... | ...  | 85     |           |

#### 詳細

**1. [バグ] ファイル名:行 — 問題の説明**（信頼度: 95）
修正案: ...

### 統計
- 検出数: X件（フィルタ前: Y件、除外: Z件）
- Agent 1（構造）: X件 / Agent 2（バグ）: X件 / Agent 3（保守性）: X件 / Agent 4（規約）: X件 / Agent 5（Codex）: X件
- クロスバリデーション: X件（Claude+Codex双方が検出）
```

**指摘がない場合:**
```
## 🔍 構造化コードレビュー結果

問題なし。5エージェント並列レビュー完了（構造・バグ・保守性・規約準拠・Codex）。
```

### 重要なルール

- **差分埋め込み方式を厳守（Agent 1〜4）**: サブエージェントがファイルを探索しに行くのを防ぐため、diffをプロンプトに直接含める。Agent 5（Codex）は `--cd` 指定で自身が diff を取得するため埋め込み不要
- **エージェントは必ず並列起動**: `--no-codex` 未指定時は5つ、指定時は4つの Agent ツールを1つのメッセージで同時に呼び出す
- **信頼度閾値は80**: 調整が必要な場合はこの数値を変更する
- **diffが500行超の場合**: ファイルごとに分割して各エージェントに渡す
- **Codex実行はサブエージェント経由**: メインBashで直接 `codex exec` を実行しない（出力が長大でコンテキストを圧迫するため）
- **Codexがタイムアウトした場合**: Agent 1〜4（Claude）の結果のみで報告する（Codexの失敗で全体を止めない）
