# Agent Teams 並列コーディング スキル

## 概要

Agent Teamsを使用して、複数エージェントが**安全に並列コーディング**を行うためのオーケストレーションスキル。
**ファイル所有権の事前設計**が核心であり、これにより上書き競合を完全に防止する。

```
                          ┌──────────────┐
                          │   リーダー   │
                          │ (delegate)   │
                          └──────┬───────┘
                                 │
                    ┌────────────┼────────────┐
                    │            │            │
              ┌─────┴─────┐ ┌───┴────┐ ┌────┴─────┐
              │ Worker A  │ │Worker B│ │ Worker C │
              │ files: a/ │ │files:b/│ │ files:c/ │
              └───────────┘ └────────┘ └──────────┘
                    │            │            │
                    ▼            ▼            ▼
              ┌─────────────────────────────────────┐
              │    排他的ファイル所有権マップ        │
              │  A: src/api/*, tests/api/*           │
              │  B: src/ui/*,  tests/ui/*            │
              │  C: src/db/*,  tests/db/*            │
              │  shared: types/ → A のみ書込可       │
              └─────────────────────────────────────┘
```

## 前提条件

### 環境変数の有効化

Agent Teams は実験的機能のため、環境変数の設定が必要。

```powershell
# PowerShellで環境変数を設定
$env:CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS = "1"

# 永続化する場合（ユーザー環境変数に追加）
[Environment]::SetEnvironmentVariable("CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS", "1", "User")
```

### 確認方法

Agent Teamsが有効かどうかは、`TeamCreate` ツールが使用可能であることで判断する。

## 核心ルール: ファイル競合防止（10箇条）

1. **1ファイル1オーナー**: 各ファイルは1人のワーカーだけが書き込み可能
2. **所有権は事前確定**: タスク開始前にファイル割当を完了する
3. **禁止ファイルの明示**: 各ワーカーに「触ってはいけないファイル」を明示する
4. **共有ファイルは single_writer**: types/、package.json等は必ず1人だけが変更権を持つ
5. **所有権移譲は3ステップ**: 通知 → 受領確認 → 編集停止（暗黙の移譲禁止）
6. **プロンプトに writable/forbidden 必須**: ワーカースポーン時に所有権範囲を明記
7. **統合は固定順序**: DB層 → 型定義 → API → UI → テスト → エントリーポイント
8. **統合前に競合検査**: 変更ファイルの重複チェックを必ず実行
9. **ワーカー上限4人**: 管理コスト > 統合利点にならないよう制限
10. **コミット前にTeamDelete禁止**: worktreeディレクトリが削除され未コミット変更が消失

---

## 7ステップ実行フロー（概要）

| Step | 名称 | 入力 | 出力 |
|------|------|------|------|
| 1 | 目的・DoD・品質ゲート定義 | ユーザーのタスク依頼 | 完了定義（DoD） |
| 2 | コードベース分析・競合ゾーン抽出 | プロジェクトディレクトリ | ファイル構造マップ |
| 3 | 所有権マップ確定 | 分析結果 | 所有権契約YAML |
| 4 | タスク分解 | DoD + 所有権マップ | TaskCreate一覧 |
| 5 | チームメイトのスポーン | タスク + 所有権 | 稼働中ワーカー |
| 6 | 監視・再割当・所有権移譲 | ワーカーからのメッセージ | 問題解決 |
| 7 | 統合 → 検証 → クリーンアップ | 完了報告 | 統合済みコード |

---

## チーム構造パターン（4種）

| パターン | 適用条件 | 推奨人数 |
|---------|---------|---------|
| フロントエンド + バックエンド分離 | API + UI が明確に分離できる場合 | 2~3人 |
| 機能モジュール分離 | 複数の独立した機能を同時実装 | 2~4人 |
| 実装 + テスト分離 | テストカバレッジを維持したい場合 | 2人 |
| リファクタリング（ディレクトリ分離） | 大規模リファクタリング | 2~3人 |

---

## クイックスタート（最小手順）

```
1. 環境変数確認
   → CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS = "1"

2. コードベースの構造を把握
   → Glob / Read でディレクトリ構造確認

3. 所有権マップ作成（最重要）
   → 所有権契約テンプレートを埋める

4. タスク作成
   → TaskCreate で依存関係・ファイル割当を明記

5. チーム作成 & ワーカースポーン
   → TeamCreate → Task tool (name, team_name, mode: "delegate")
   → プロンプトに writable / forbidden を必ず記載

6. 監視 & 完了報告収集
   → TaskList で進捗確認、SendMessage で調整

7. 統合 & 検証
   → 基盤層から順に統合 → テスト → クリーンアップ
```
