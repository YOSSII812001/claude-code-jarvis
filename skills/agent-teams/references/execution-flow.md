# Agent Teams 実行フロー詳細

## 実行フロー（7ステップ）

### Step 1: 目的・DoD・品質ゲート定義

**入力**: ユーザーからのタスク依頼
**出力**: 明確な目的、完了定義（DoD）、品質ゲート
**完了条件**: ユーザーがDoDに合意

```
目的の定義例:
  目的: ユーザー認証機能をフロントエンド・バックエンド・DB層で並列実装
  DoD（完了定義）:
    - [ ] APIエンドポイント（signup, login, logout）が動作
    - [ ] UIにログインフォームが表示される
    - [ ] DBにユーザーテーブルが作成される
    - [ ] 全コンポーネントが型安全に接続される
    - [ ] テストが通る
  品質ゲート:
    - TypeScriptの型エラーがゼロ
    - ESLintエラーがゼロ
    - テストカバレッジ > 80%
```

### Step 2: コードベース分析と競合ゾーン抽出

**入力**: プロジェクトディレクトリ
**出力**: ファイル構造マップ、競合ゾーン一覧
**完了条件**: 全ての共有ファイルと競合ゾーンが特定されている

```
分析に使用するツール:
  - Glob: ディレクトリ構造の把握
    例: Glob("src/**/*.ts"), Glob("**/*.config.*")
  - Grep: importの依存関係分析
    例: Grep("import.*from", type: "ts")
  - Read: エントリーポイントと共有ファイルの内容確認

チェック項目:
  [] ディレクトリ構造の全体把握
  [] エントリーポイントの特定（index.ts, App.tsx等）
  [] 共有型定義ファイルの特定
  [] 設定ファイルの特定
  [] パッケージ依存関係の確認
  [] テストファイルの配置パターン確認
  [] importの循環依存がないか確認
```

### Step 3: 所有権マップ確定

**入力**: Step 2 の分析結果
**出力**: 所有権契約（所有権契約テンプレート形式）
**完了条件**: 全ファイルに対してオーナーまたは「変更禁止」が割り当てられている

```
手順:
  1. 「所有権契約テンプレート」を使用してYAMLを作成
  2. 各ワーカーの writable / forbidden を定義
  3. shared_files の single_writer を決定
  4. 境界が曖昧なファイルは「変更禁止」にして事前に完了させる
  5. 所有権マップをユーザーに提示して合意を得る
```

### Step 4: タスク分解

**入力**: Step 1 の DoD + Step 3 の所有権マップ
**出力**: TaskCreateで作成されたタスク一覧（依存関係・ファイル割当明記）
**完了条件**: 全タスクにファイル割当と依存関係が設定されている

```
TaskCreate の記述ルール:
  - subject: 簡潔なタスク名（命令形）
  - description に必ず以下を含める:
    - 担当ファイル（writableパス）
    - 禁止ファイル（forbiddenパス）
    - 依存タスク（blockedBy）
    - 完了条件
  - activeForm: 進行形の動詞で記述

例:
  TaskCreate:
    subject: "認証APIエンドポイントを実装"
    description: |
      ## 書込可能ファイル
      - src/api/auth.ts（新規作成）
      - src/api/middleware.ts（新規作成）
      - tests/api/auth.test.ts（新規作成）

      ## 禁止ファイル（絶対に編集しない）
      - src/ui/**
      - src/db/**
      - package.json
      - types/**（worker-c が担当）

      ## 完了条件
      - signup, login, logout の3エンドポイントが実装済み
      - テストファイルが作成済み
    activeForm: "認証APIエンドポイントを実装中"
```

### Step 5: チームメイトのスポーン

**入力**: Step 4 のタスク一覧 + Step 3 の所有権マップ
**出力**: 稼働中のチームメイト
**完了条件**: 全チームメイトが自分の所有権範囲を理解してタスクに着手

```
TeamCreate:
  team_name: "feature-auth"
  description: "ユーザー認証機能の並列実装"

Task tool（各ワーカーのスポーン）:
  subagent_type: "general-purpose"
  team_name: "feature-auth"
  name: "worker-api"
  mode: "delegate"
  prompt: |
    あなたは認証APIの実装担当です。

    ## 所有権ルール（厳守）
    書込可能:
      - src/api/**
      - tests/api/**
    禁止（絶対に触るな）:
      - src/ui/**
      - src/db/**
      - types/**
      - package.json

    ## タスク
    [具体的なタスク内容]

    ## 完了時
    1. 変更したファイル一覧を報告
    2. テスト結果を報告
    3. TaskUpdate で完了マーク
```

**重要**: プロンプトに「書込可能パス」と「禁止パス」を**必ず**含める。

### Step 6: 監視・再割当・所有権移譲処理

**入力**: 稼働中のチームメイトからのメッセージ
**出力**: 問題の解決、必要に応じた所有権移譲
**完了条件**: 全タスクが完了し、競合なし

```
リーダーの監視項目:
  [] 各ワーカーのタスク進捗（TaskList で確認）
  [] ワーカー間の依存関係の解消
  [] 所有権移譲リクエストへの対応
  [] ブロッカーの特定と解消
  [] 予期しないファイル変更の検出

所有権移譲が必要なケース:
  - Worker A の実装で Worker B の担当ファイルに変更が必要と判明
  - タスクの依存関係が変更された
  → 所有権移譲プロトコル（ルール5）に従って処理

ワーカーが完了報告時に含めるべき情報:
  1. 変更したファイルの一覧
  2. 新規作成したファイルの一覧
  3. テスト実行結果
  4. 未解決の問題（あれば）
```

### Step 7: 統合 → 検証 → クリーンアップ

**入力**: 全ワーカーの完了報告
**出力**: 統合されたコード、検証結果
**完了条件**: DoDの全項目がチェック済み

#### Step 7a: 統合（固定順序）

```
統合順序（推奨）:
  1. DB / スキーマ層（最も基盤）
  2. 共有型定義 / インターフェース
  3. バックエンド / API層
  4. フロントエンド / UI層
  5. テスト
  6. エントリーポイント / ルーティング

理由: 依存される側から統合することで、型エラーを段階的に解消できる
```

#### Step 7b: 自動検証

```
検証チェックリスト:
  [] 統合前競合検査（ルール6のスクリプト実行）
  [] TypeScript型チェック（tsc --noEmit）
  [] ESLint / リンター実行
  [] テスト実行（全テスト）
  [] ビルド成功確認
  [] DoDの全項目チェック
```

#### Step 7c: クリーンアップ

```
クリーンアップ項目:
  [] 不要なTODOコメントの削除
  [] console.log / デバッグコードの削除
  [] 未使用importの削除
  [] チームのシャットダウン（SendMessage type: "shutdown_request"）
  [] TeamDelete でチームリソース解放
```

## 所有権契約テンプレート

タスク開始時にリーダーが作成し、各ワーカーのプロンプトに含める。

```yaml
# === 所有権契約 ===
ownership:
  worker-a:
    writable:                    # 書き込み可能なパス
      - "src/api/**"
      - "tests/api/**"
      - "src/lib/api-utils.ts"
    forbidden:                   # 絶対に触ってはいけないパス
      - "src/ui/**"
      - "src/db/**"
      - "package.json"
      - "types/**"

  worker-b:
    writable:
      - "src/ui/**"
      - "tests/ui/**"
      - "src/components/**"
    forbidden:
      - "src/api/**"
      - "src/db/**"
      - "package.json"
      - "types/**"

  worker-c:
    writable:
      - "src/db/**"
      - "tests/db/**"
      - "prisma/schema.prisma"
    forbidden:
      - "src/api/**"
      - "src/ui/**"
      - "package.json"

# 共有ファイル管理
shared_files:
  - file: "types/index.ts"
    single_writer: "worker-a"     # このファイルを書けるのは worker-a だけ
    reason: "API型定義が起点のため"

  - file: "package.json"
    single_writer: "leader"       # リーダーが統合時に一括管理
    reason: "依存関係の競合防止"

  - file: "src/index.ts"
    single_writer: "worker-a"
    reason: "エントリーポイントの一貫性"

# 所有権移譲ルール
handoff:
  required_fields:
    - reason          # なぜ移譲が必要か
    - files           # 対象ファイル一覧
    - receiver_ack    # 受領者の確認（必須）
  protocol: |
    1. 移譲元 → SendMessage（理由 + ファイル一覧）
    2. 移譲先 → SendMessage（受領確認 + ファイル復唱）
    3. 移譲元 → 当該ファイルの編集停止
```

## タスクサイズガイドライン

| サイズ | ファイル数目安 | 所要時間目安 | 適切なケース |
|--------|--------------|-------------|-------------|
| **小** | 1~3ファイル | 短い | 単一機能の追加、バグ修正 |
| **中** | 4~10ファイル | 中程度 | 機能モジュール実装、APIエンドポイント群 |
| **大** | 10ファイル超 | 長い | フルスタック機能、大規模リファクタリング |

```
チーム推奨サイズ:
  小タスク → チーム不要（単独で十分）
  中タスク → 2人チーム
  大タスク → 2~4人チーム（4人が上限推奨）

注意:
  - ワーカーが多すぎると管理コストが統合の利点を上回る
  - 2人で開始し、必要に応じて追加するのが安全
  - 各ワーカーのタスクは「中」サイズ以下に分解する
```
