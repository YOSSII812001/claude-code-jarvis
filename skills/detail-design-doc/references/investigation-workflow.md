# コードベース調査ワークフロー

> Codex CLI + design-review-checklist を統合した調査フロー。
> issue-planner の Step 2-3.5 をベースに、設計書向けに拡張。

---

## Step 2a: Codex CLI 調査

### コマンドパターン（stdin経由、read-only）

issue-planner と同じパターンで Codex CLI を実行する:

```bash
cat > /tmp/design-prompt.txt << 'PROMPT_EOF'
{プロンプト内容}
PROMPT_EOF
cat /tmp/design-prompt.txt | codex exec \
  --mode read-only \
  --no-mcp \
  --full-stdout \
  -q \
  - 2>/dev/null || echo "CODEX_FAILED"
rm -f /tmp/design-prompt.txt
```

### 設計向けプロンプト（8項目分析）

入力種別に応じてコンテキストを調整した上で、以下の8項目を分析させる:

```
このプロジェクトのコードベースを調査し、以下の設計対象について分析してください。

【設計対象】
{入力種別に応じた要件記述}

【分析項目】
1. **関連コード**: 設計対象に関連する既存ファイル・モジュールをリストアップ（パス付き）
2. **アーキテクチャパターン**: プロジェクトで使用されているアーキテクチャパターン（ディレクトリ構成、レイヤー分離、命名規則）
3. **DB構造**: 関連するテーブル定義、RLSポリシー、マイグレーション履歴
4. **API構造**: 関連するAPIエンドポイント、ルーティングパターン、ミドルウェア
5. **コンポーネント構造**: 関連するUIコンポーネント、共通コンポーネント、レイアウト
6. **状態管理**: 使用されている状態管理パターン（React Query, useState, Context等）
7. **類似実装**: 設計対象に類似する既存機能の実装パターン（最も参考になるもの）
8. **技術制約**: 使用ライブラリのバージョン制約、既知の制限事項

各項目について、具体的なファイルパスとコード断片を含めて回答してください。
```

### 入力種別によるプロンプト調整

| 入力種別 | コンテキスト埋め込み |
|---------|-------------------|
| Issue入力 | Issue本文を要約して500文字以内で埋め込み: `【設計対象】Issue #{number}: {title}\n{body_summary}` |
| .md仕様書入力 | 仕様書の「要件」「ユースケース」セクションを抽出して1000文字以内で埋め込み |
| テキスト要件 | ユーザーの要件記述をそのまま埋め込み |

### フォールバック（4段階、issue-planner踏襲）

| 段階 | 条件 | 対応 |
|------|------|------|
| 1 | Codex実行失敗 / タイムアウト(180秒) | プロンプトを200文字に圧縮して再実行 |
| 2 | 圧縮版も失敗 | `--reasoning-effort medium` を追加 |
| 3 | それでも失敗 | `--skip-git-repo-check` を追加 |
| 4 | 全て失敗 | Codex調査をスキップし、Glob/Grep/Read で手動調査に切り替え |

フォールバック段階4の場合、以下の手動調査を実施:
1. `Glob` で関連ファイルパターンを検索
2. `Grep` でキーワード検索（関連する関数名、テーブル名、コンポーネント名）
3. `Read` で主要ファイルの内容を確認
4. 調査結果を Codex 出力と同じ構造で整理

## Step 2b: design-review-checklist 統合

設計内容に応じて、以下のチェック項目を **選択的に** 実行する。
全項目必須ではなく、設計対象のスコープに応じて該当するものだけを実施。

### 常時実行（全設計対象）

#### Phase 1.1: データフロー全経路の洗い出し
```
対象ファイルに関連するデータの更新経路を全て特定:
- SSE（リアルタイム通知）
- REST ポーリング（定期取得）
- REST 同期実行（即座の取得）
- 確認必須フロー
- エラー/キャンセルフロー
- 初回取得フロー
```

#### Phase 1.2: ペイロード構造確認
```
関連するAPI/SSEイベントのペイロード構造を確認:
- emitEvent() 呼び出し箇所
- TypeScript型定義
- 実際の実装コードでのフィールド確認
```

#### Phase 1.3: 既存型定義確認
```
新規型定義の前に既存型を確認:
- 同名・類似の型が既に存在しないか
- 既存型を拡張すべきか、新規作成すべきか
```

### UI設計時のみ実行

#### Phase 1.4: 既存UIの分岐ロジック
```
モバイル/タブレット/デスクトップの display 分岐:
- メディアクエリのブレークポイント
- useMediaQuery の使用パターン
- 条件付きレンダリングのパターン
```

#### Phase 1.5: CSSグローバルルール
```
!important 付きルールとの競合確認:
- グローバルCSSの !important ルール
- テーマ設定との整合性
- CSS Module / styled-components のスコーピング
```

### DB設計時のみ実行

#### Phase 4: DB・マイグレーション安全性
```
マイグレーション安全性チェック:
- DROP COLUMN / DROP POLICY の有無
- 手動追加カラムの保持確認
- 空文字列 vs NULL の取り扱い統一
- TIMESTAMPTZ の使用確認（TIMESTAMP禁止）
- 既存RLSポリシーとの整合性
```

### API設計時のみ実行

#### Phase 5: ハンドラ全数列挙
```
マルチハンドラファイルの網羅性:
- 対象ファイルの全ハンドラ（export された関数）を列挙
- フロントエンド → バックエンド呼び出し経路を grep で確認
- 未使用ハンドラの特定
```

### 実行判定マトリクス

| 設計内容 | 1.1 | 1.2 | 1.3 | 1.4 | 1.5 | Phase 4 | Phase 5 |
|---------|-----|-----|-----|-----|-----|---------|---------|
| フルスタック新機能 | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ |
| バックエンドAPI追加 | ✓ | ✓ | ✓ | - | - | ✓ | ✓ |
| フロントエンドUI追加 | ✓ | ✓ | ✓ | ✓ | ✓ | - | - |
| DB変更のみ | ✓ | - | ✓ | - | - | ✓ | - |
| リファクタリング | ✓ | - | ✓ | - | - | - | ✓ |

## Step 2c: コード現状検証（issue-planner Step 3.5 踏襲）

### 目的
変更対象ファイルの最新状態を確認し、設計前提との乖離を検出する。

### 手順
1. 変更対象ファイルのリストアップ（Step 2a の結果から）
2. 各ファイルに対して `git log -5 --oneline -- {file_path}` を実行
3. 最近の変更がある場合、設計前提に影響がないか確認

### 複雑度分類
| 分類 | 基準 | 設計への影響 |
|------|------|-------------|
| 低 | 単純な値変更、1テーブル操作 | 最小限の調査で設計可能 |
| 中 | 依存パラメータ、ファクトリ関数、複数コンポーネント連携 | 関連コードの詳細調査が必要 |
| 高 | 複数テーブル並列fetch、Promise.allSettled、複合ロジック | 網羅的な調査 + 類似実装の参照が必須 |

### チェック項目
1. **他Issueによる最近変更**: 直近2週間で変更されたファイルに設計が依存していないか
2. **既に実装済みでないか**: 設計対象の機能が部分的に実装されていないか
3. **データフェッチパターンの複雑度**: 既存のフェッチパターンと新規設計の整合性
