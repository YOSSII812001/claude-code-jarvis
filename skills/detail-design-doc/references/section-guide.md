# セクション記述ガイド + ADRテンプレート

> 各セクションの「何を書くか」「何を書かないか」「N/A判断基準」を定義する。
> 設計書の品質均一化と、N/Aセクションの理由明示（核心ルール#4）を支援する。

---

## セクション別ガイド

### 1. 概要
- **目的**: 設計の背景・目的・スコープを簡潔に伝え、読み手が設計書全体の文脈を理解できるようにする
- **書くべきこと**: なぜこの設計が必要か、何を達成するか、何を含み何を含まないか
- **書かないこと**: 技術的な詳細（それは後続セクションで）
- **N/A判断**: このセクションはN/A不可（必須）
- **記述例**:
  ```
  ### 1.1 背景
  現在の検索機能はフルテーブルスキャンに依存しており、データ量増加に伴い
  レスポンスタイムが3秒を超えるケースが発生している。
  ```

### 2. 要件定義
- **目的**: 実装すべき機能と品質基準を明確化し、完了条件を定義する
- **書くべきこと**: 機能要件（FR-XX）と非機能要件（NFR-XX）、それぞれにIDと受入条件を付与
- **書かないこと**: 実装方法（HOWではなくWHATを書く）
- **N/A判断**: このセクションはN/A不可（必須）
- **入力種別ごとの注意**:
  - Issue入力: Issue本文から要件を抽出・構造化。不明確な要件は「要確認」マーク
  - .md仕様書: 仕様書の要件セクションをそのまま活用、ID体系を統一
  - テキスト要件: 曖昧な記述を具体化、抜けている非機能要件を補完

### 3. アーキテクチャ設計
- **目的**: システム全体の構造と、既存アーキテクチャとの関係を示す
- **書くべきこと**: コンポーネント構成、データフロー、既存パターンとの整合性
- **書かないこと**: 個別コンポーネントの内部実装詳細
- **N/A判断**: 既存コンポーネント内の小規模変更で、アーキテクチャに影響がない場合
  - N/A例: 「既存コンポーネント内のバリデーションロジック追加のため、アーキテクチャレベルの変更なし」

### 4. DB設計
- **目的**: データモデルの変更を明確化し、マイグレーションの安全性を保証する
- **書くべきこと**: CREATE TABLE / ALTER TABLE の DDL、RLSポリシー、インデックス、既存スキーマとの整合性
- **書かないこと**: クエリ最適化の詳細（それはパフォーマンスチューニング時に）
- **N/A判断**: DB変更を伴わない場合
  - N/A例: 「フロントエンドのみの変更のため、DB設計はN/A」
- **必須チェック**:
  - TIMESTAMPTZ の使用（TIMESTAMP禁止）
  - DROP COLUMN / DROP POLICY がある場合は影響範囲を明記
  - 手動追加カラムの保持確認

### 5. API設計
- **目的**: フロントエンド・バックエンド間の契約を定義する
- **書くべきこと**: エンドポイント一覧、リクエスト/レスポンスの型定義、エラーレスポンス
- **書かないこと**: API内部の実装ロジック
- **N/A判断**: 新規API追加も既存API変更もない場合
  - N/A例: 「既存APIの利用のみで、API変更なし」
- **注意**: 既存ルーティングパターンとの整合性を確認（design-review-checklist Phase 5）

### 6. UI設計
- **目的**: ユーザーインターフェースの構造と挙動を定義する
- **書くべきこと**: 画面遷移、コンポーネントツリー、レスポンシブ対応
- **書かないこと**: CSSの具体的なプロパティ値（重要な制約を除く）
- **N/A判断**: バックエンドのみの変更の場合
  - N/A例: 「バッチ処理の追加のため、UI変更なし」
- **注意**: 既存UIの分岐ロジック確認（design-review-checklist Phase 1.4, 1.5）

### 7. 状態管理設計
- **目的**: アプリケーション状態の管理方法と遷移を明確化する
- **書くべきこと**: 状態の分類（サーバー/UI/フォーム）、管理方法、遷移図
- **書かないこと**: 個別のuseState変数レベルの詳細
- **N/A判断**: 状態管理に変更がない場合、またはバックエンドのみの変更
  - N/A例: 「既存の状態管理パターンをそのまま使用し、新規状態の追加なし」
- **注意**: 状態遷移はバックエンドの実コードで確認（推測しない）

### 8. エラーハンドリング設計
- **目的**: エラー発生時の挙動を統一的に定義する
- **書くべきこと**: エラー分類、ユーザー表示方法、リトライ戦略
- **書かないこと**: 個別のtry-catchの書き方
- **N/A判断**: 既存のエラーハンドリングパターンをそのまま踏襲する場合
  - N/A例: 「既存のグローバルエラーハンドラーを使用し、カスタムエラー処理は不要」

### 9. テスト戦略
- **目的**: テストの範囲と方法を定義し、品質を保証する
- **書くべきこと**: テスト種別、対象、ツール、テストケースID
- **書かないこと**: テストコードの全文
- **N/A判断**: このセクションはN/A不可（必須。最低限E2Eテストケースを定義する）

### 10. セキュリティ考慮事項
- **目的**: セキュリティリスクと対策を明示する
- **書くべきこと**: 認証認可、入力検証、データ保護
- **書かないこと**: セキュリティツールの設定詳細
- **N/A判断**: セキュリティに影響のない純粋なUI改善の場合（ただし慎重に判断）
  - N/A例: 「表示テキストの文言変更のみで、データの入出力に変更なし」

### 11. 実装ステップ
- **目的**: 実装の順序と優先度を示し、段階的に進められるようにする
- **書くべきこと**: Phase分け、各ステップの対象ファイルと概要
- **書かないこと**: 具体的なコード変更内容（影響範囲セクションで扱う）
- **N/A判断**: このセクションはN/A不可（必須）

### 12. リスクと軽減策
- **目的**: 実装リスクを事前に特定し、対策を準備する
- **書くべきこと**: リスクID、内容、確率、影響度、軽減策
- **書かないこと**: リスクが顕在化した後の対応手順（それはインシデント対応）
- **N/A判断**: このセクションはN/A不可（必須。リスクがない設計はない）

### 13. 設計判断記録（ADR）
- **目的**: 設計上の重要な判断とその根拠を記録し、将来の参照に備える
- **書くべきこと**: ADR-XXX形式で、コンテキスト、選択肢比較、決定、根拠
- **書かないこと**: 自明な判断（既存パターンをそのまま踏襲する場合等）
- **N/A判断**: このセクションはN/A不可（核心ルール#2: ADR必須）
- **最低1件**: 設計に判断が含まれない場合でも、「技術選定」や「既存パターン踏襲の判断」をADRとして記録

### 14. 影響範囲
- **目的**: 変更対象ファイルと現状との乖離を明確化する
- **書くべきこと**: ファイル一覧（変更種別・複雑度付き）、git logによる現状検証結果
- **書かないこと**: ファイル内容の全文（インターフェース定義と重要ロジックのみ）
- **N/A判断**: このセクションはN/A不可（必須）

## ADRテンプレート

設計判断が発生した場合、以下のテンプレートで記録する:

```markdown
### ADR-XXX: {判断タイトル}

- **ステータス**: 承認 / 検討中 / 却下
- **コンテキスト**: {判断が必要になった背景。なぜ複数の選択肢があるのか}
- **選択肢**:
  - A) {選択肢A}
    - メリット: {メリット1}, {メリット2}
    - デメリット: {デメリット1}, {デメリット2}
    - 工数: {S/M/L}
  - B) {選択肢B}
    - メリット: {メリット1}, {メリット2}
    - デメリット: {デメリット1}, {デメリット2}
    - 工数: {S/M/L}
  - C) {選択肢C}（必要に応じて）
    - メリット: {メリット1}
    - デメリット: {デメリット1}
    - 工数: {S/M/L}
- **決定**: {選択した案（例: B案を採用）}
- **根拠**: {なぜその案を選んだか。トレードオフの評価基準を含む}
- **影響**: {この決定による後続の設計・実装への影響}
```

### ADR記述のベストプラクティス
1. **コンテキストは具体的に**: 「パフォーマンスが重要」ではなく「1000件以上のレコードで3秒以内のレスポンスが必要（NFR-01）」
2. **選択肢は最低2つ**: 1つしかない場合は判断ではない
3. **根拠にデータを含む**: 「Bの方が速い」ではなく「ベンチマークでBはAの3倍のスループット」
4. **影響は具体的に**: 「他の機能に影響」ではなく「検索APIのレスポンス型が変更されるため、SearchResults.tsxの修正が必要」

## コード貼りすぎ防止ガイドライン（核心ルール#5）

設計書に含めるコード:
- **含める**: インターフェース定義、型定義、DDL、重要なロジックの擬似コード
- **含めない**: 完全な実装コード、テストコードの全文、設定ファイルの全文

目安:
- 1セクションあたりのコードブロック: 最大3つ
- 1コードブロックの行数: 最大30行
- 全体のコード比率: 設計書全体の30%以下
