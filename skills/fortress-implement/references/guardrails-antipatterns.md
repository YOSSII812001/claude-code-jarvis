# fortress-implement — ガードレール & アンチパターン

## ガードレール（15項目）

### Phase 0 ゲート

| # | ガードレール | 検証方法 | 違反時のアクション |
|---|------------|---------|------------------|
| G1 | 曖昧な受入条件のまま実装開始しない | 凍結チェックリスト全項目確認 | Phase 0で停止、ユーザーに明確化を依頼 |
| G2 | Mission Briefが未承認のままPhase 1に進まない | ユーザー承認確認 | 承認待ち |

### Phase 1 ゲート

| # | ガードレール | 検証方法 | 違反時のアクション |
|---|------------|---------|------------------|
| G3 | fortress-review未実施のままI2+タスクの実装を開始しない | Tier判定 + fortress-reviewステータス | Phase 1で停止 |
| G4 | 独立ロールバック不可能なSliceを計画に含めない | Slice分解の依存関係チェック | Slice再分解を強制 |

### Phase 2 ゲート（Slice単位）

| # | ガードレール | 検証方法 | 違反時のアクション |
|---|------------|---------|------------------|
| G5 | テスト先行なしの実装を許可しない | Step A完了確認 | Step Aに戻る |
| G6 | 1 Sliceにリファクタ + 機能追加を混在させない | diff審査 | Slice分割を強制 |
| G7 | 想定外ファイル差分を検知したら即停止 | `git diff --stat` vs Slice計画の予定ファイル | 差分の確認をユーザーに依頼 |
| G8 | クロスチェック未完了で次Sliceに進まない | Step C完了確認 | Step C実行を強制 |
| G9 | lint/type/test未PASSでSafe Pointを作成しない | Step D実行結果 | Self-Healing Loop起動 |
| G10 | Safe Point作成を省略しない | Step E実行確認 | 強制作成 |
| G11 | Self-Healing 3回失敗でユーザーエスカレーション | failure_count確認 | ロールバック + ユーザー報告 |
| G12 | flakyテストをPASS扱いしない | テスト結果の安定性確認（2回実行） | flakyとして分類・隔離 |

### Phase 3-4 ゲート

| # | ガードレール | 検証方法 | 違反時のアクション |
|---|------------|---------|------------------|
| G13 | 統合テスト全PASSなしにPhase 4に進まない | テスト実行結果 | Phase 3でループ |
| G14 | 証跡パックが不完全なまま完了判定しない | パック内容チェック | 不足項目を補完 |
| G15 | CRITICAL/HIGH未解決でGo判定を出さない | 指摘ステータス確認 | No-Go or 条件付きGo |

## アンチパターン（15項目）

| # | アンチパターン | なぜ危険か | 防止策（対応ガードレール） |
|---|-------------|----------|------------------------|
| AP1 | 一括実装して最後にまとめてレビュー | 問題の発見が遅れ、手戻りが巨大化 | Slice単位の段階的実装（G5-G10） |
| AP2 | コードを書いてから仕様を説明する | 仕様と実装の乖離を事後正当化 | Phase 0の要件凍結（G1, G2） |
| AP3 | 実装者が自分のコードだけで完了判定 | 自己正当化バイアス | 実装者以外のクロスチェック（G8） |
| AP4 | failing testなしのbugfix | 修正の正当性を証明できない | テスト先行の強制（G5） |
| AP5 | テストを削除/無効化してGREENにする | 問題を隠蔽 | diff監査で検出、即停止（G7） |
| AP6 | 不安定なテストを「たまたま通った」で済ませる | 本番で再発するリスク | flakyテスト分類・隔離（G12） |
| AP7 | Safe Pointを省略して数Slice連続で進む | 障害時のロールバック不能 | Safe Point作成の強制（G10） |
| AP8 | 影響範囲の広い変更をTier低く見積もる | 検証が不十分になる | シグナルテーブルによる客観的判定 |
| AP9 | 依存更新を機能実装Sliceに紛れ込ませる | 副作用の切り分け不能 | 関心事混在の禁止（G6） |
| AP10 | 設計前提が崩れたのに計画を更新せず続行 | 前提不成立の上に実装を積み重ね | Safe Point前提メモで毎回検証 |
| AP11 | 単体テストPASSだけで「完了」と判断 | 統合問題・E2E問題を見逃す | Phase 3の統合検証（G13） |
| AP12 | 証跡を残さず口頭要約だけで完了とする | 後から検証・再現不能 | 証跡パック必須（G14） |
| AP13 | fortress-reviewのCRITICAL指摘を無視して実装開始 | 設計レベルの問題が実装に波及 | fortress-review必須ゲート（G3） |
| AP14 | N-version結果の不一致を無視する | 重要ロジックの正当性が未確認 | 不一致箇所の裁定を強制（I3専用） |
| AP15 | Self-Healing中に根本原因を調査せず再試行 | 同じ失敗を繰り返す | 同一パターン2回で別アプローチ強制（G11） |

## ガードレール違反時のエスカレーション

| 違反レベル | 条件 | アクション |
|-----------|------|----------|
| **自動解決** | G5, G6, G10（手順スキップ） | 不足ステップに自動で戻る |
| **Self-Healing** | G9（テスト失敗） | Self-Healing Loop起動 |
| **ユーザー確認** | G1, G2, G7（曖昧・想定外） | ユーザーに判断を仰ぐ |
| **強制停止** | G3, G11, G15（安全弁） | 実装を停止し、問題を報告 |
