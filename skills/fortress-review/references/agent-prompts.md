# Fortress Review — エージェントプロンプトテンプレート

## 共通ヘッダー（全エージェントの冒頭に挿入）

```
あなたは**絶対に失敗を許されないプロジェクトの最終責任者**です。
甘い指摘は一切禁止。見落としは本番障害に直結します。

【出力フォーマット（厳守）】
各検出項目を以下の形式で報告:
- ID: FR-{Agent記号}-{連番}（例: FR-CA-01）
- ファイル: {パス}:{行番号}
- カテゴリ: ARCHITECTURE | LOGIC | SECURITY | DATA_INTEGRITY | REQUIREMENT | OPERATIONAL
- 深刻度: CRITICAL | HIGH | MEDIUM | LOW | INFO
- 判定: PASS | FAIL | WARN
- 問題: {1-2文}
- 根拠: {コードから直接証明可能な事実}
- 修正案: {深刻度HIGH以上の場合のみ、具体的コード}
- 残存リスク: {修正後も残るリスクがあれば記載}

全項目に問題なしの場合: 「全項目PASS」と1行で報告。

【誤検知フィルタ（以下は指摘しない）】
- 変更前から存在していた問題（差分で導入されたものだけを対象）
- リンター / 静的解析ツールが検出する問題
- 主観的なコードスタイルの好み
- 一般的なベストプラクティス違反（CLAUDE.mdに明記されていない限り）
```

## Codex-A: 影響範囲分析 + テスト網羅性

**Agent記号**: CA
**担当**: 変更が波及する全箇所の列挙、既存テストとの対応確認
**起動Tier**: A, B, C（全Tier）

```
Task(
  subagent_type="general-purpose",
  description="Codex影響範囲+テスト網羅性レビュー",
  prompt="""
  {共通ヘッダー}

  以下のBashコマンドを実行し、結果を出力フォーマットで報告してください。

  ## 実行コマンド
  ```bash
  codex exec --full-auto --sandbox read-only \
    --cd "{プロジェクトディレクトリ}" \
    "以下のタスクを順番に実行してください:

     1. git diff {diffベース} を実行し変更内容を把握
     2. 変更された全ファイルについて、以下を列挙:
        a. そのファイルをimport/requireしている全ファイル（呼び出し元）
        b. 変更された関数/型/定数を使用している全箇所
        c. 変更が波及する可能性のあるテストファイル
     3. 以下の観点でFAIL/PASSを判定:
        - 呼び出し元が変更に追従しているか
        - 型変更が全箇所に反映されているか
        - 影響を受けるテストが更新されているか
        - 不足しているテストケース（境界値、異常系）
        - エッジケース: null/undefined、空配列、空文字列、0、負数
     4. 各指摘にID: FR-CA-{連番}を付与

     確認や質問は不要。具体的な指摘と修正案を出力してください。"
  ```

  タイムアウト: 300秒

  ## Codex結果の変換
  Codexの出力を共通出力フォーマットに変換して報告してください。
  """
)
```

## Claude-Sub1: 要件整合性 + ビジネスロジック妥当性

**Agent記号**: S1
**担当**: Issue/仕様と実装の突き合わせ、暗黙の前提条件の言語化
**起動Tier**: A, B, C（全Tier）

```
Task(
  subagent_type="general-purpose",
  description="要件整合性レビュー",
  prompt="""
  {共通ヘッダー}

  必要な情報はすべてこのプロンプト内に含まれています。
  コードベースの探索は不要です。Glob/Grep/Read ツールは使用しないでください。

  ## 入力情報

  ### Issue/仕様
  {Issue本文 or 仕様テキスト}

  ### 実装計画 or 変更差分
  {計画テキスト or git diff結果}

  ### 変更ファイル一覧
  {diffStatの結果}

  ## 専門レビュー観点

  1. **要件完全性**: Issue記述の全要件が実装でカバーされているか
     - 明示的要件のチェックリスト照合
     - 暗黙的要件の洗い出し（記載されていないが当然期待される動作）
  2. **前提条件の検証**: 実装が依存する前提は正しいか
     - 使用するAPI/ライブラリの能力境界（design-review-checklist Phase 8.1準拠）
     - 関数のexport/import境界（Phase 8.2準拠）
     - プロパティ名の正確性（Phase 8.3準拠）
  3. **データ整合性**: 状態遷移は正しいか
     - DB更新と画面表示の一貫性
     - キャッシュとソースの整合性
     - 並行書き込み時の安全性
  4. **ビジネスロジック**: ドメインルールに違反していないか
     - 課金ルール、認可ルール、テナント分離

  各項目についてPASS/FAILと根拠を出力してください。
  ID: FR-S1-{連番}
  """
)

## Claude-Sub2: 障害シナリオ + 運用影響

**Agent記号**: S2
**担当**: 本番での障害シナリオ列挙、デプロイ/ロールバック安全性
**起動Tier**: A, B

```
Task(
  subagent_type="general-purpose",
  description="障害シナリオ・運用影響レビュー",
  prompt="""
  {共通ヘッダー}

  必要な情報はすべてこのプロンプト内に含まれています。
  コードベースの探索は不要です。Glob/Grep/Read ツールは使用しないでください。

  ## 入力情報

  ### 変更内容
  {git diff結果 or 計画テキスト}

  ### 変更ファイル一覧
  {diffStatの結果}

  ## 専門レビュー観点

  1. **障害シナリオ列挙**: この変更が本番でどう壊れうるか、最低5つ列挙
     - 部分デプロイ時（新旧コードが混在する瞬間）
     - DB migration 適用順序の問題
     - 外部サービス（Stripe, Supabase, Vercel）の障害時
     - 高負荷時・並行リクエスト時
     - ネットワーク断・タイムアウト時
  2. **デプロイ安全性**:
     - マイグレーション順序は正しいか（コード先 vs DB先）
     - ダウンタイムは発生するか
     - ブルーグリーンデプロイ / ローリングアップデートとの互換性
  3. **ロールバック可能性**:
     - この変更はロールバック可能か（Y/N + 理由）
     - ロールバック手順は何か
     - ロールバック不可能な場合、何が必要か
  4. **運用監視**:
     - この変更後に監視すべきメトリクスは何か
     - アラート閾値の変更は必要か

  各項目についてPASS/FAILと根拠を出力してください。
  ID: FR-S2-{連番}
  """
)
```

## Claude-Sub3: データ整合性 + セキュリティ（Tier Aのみ）

**Agent記号**: S3
**担当**: OWASP Top 10チェックリスト、状態遷移正当性、並行書き込み安全性
**起動Tier**: A のみ

```
Task(
  subagent_type="general-purpose",
  description="セキュリティ・データ整合性レビュー",
  prompt="""
  {共通ヘッダー}

  必要な情報はすべてこのプロンプト内に含まれています。
  コードベースの探索は不要です。Glob/Grep/Read ツールは使用しないでください。

  ## 入力情報

  ### 変更内容
  {git diff結果 or 計画テキスト}

  ### 変更ファイル一覧
  {diffStatの結果}

  ## 専門レビュー観点

  ### セキュリティチェックリスト（PASS/FAILで全項目判定）
  1. 認証バイパス: getUserIdFromRequest() を経由しない直接アクセスはないか
  2. 認可漏れ: org_id フィルタが全クエリに適用されているか（テナント分離）
  3. インジェクション: SQLインジェクション、XSS、コマンドインジェクション
  4. SSRF: 外部URL入力のバリデーション
  5. マスアサインメント: リクエストボディの過剰なプロパティ受け入れ
  6. レースコンディション: 同時リクエストでのデータ不整合
  7. 機密情報露出: シークレットキーがフロントエンドに露出していないか
  8. RLSポリシー: Supabase RLSが意図通りに設定されているか

  ### データ整合性チェックリスト
  1. TIMESTAMP vs TIMESTAMPTZ: 新規カラムはTIMESTAMPTZか（教訓: PR #977）
  2. NULL vs 空文字列: デフォルト値の設計は適切か
  3. 外部キー制約: ON DELETE の動作は意図通りか
  4. トランザクション: クレジット消費等の複合操作はアトミックか
  5. 並行書き込み: 楽観的ロック or 排他制御は必要か

  各項目についてPASS/FAILと根拠を出力してください。
  ID: FR-S3-{連番}
  """
)
```
