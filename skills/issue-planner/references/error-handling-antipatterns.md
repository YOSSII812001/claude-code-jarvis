# エラーハンドリング、アンチパターン

## スキップ判定基準

以下のいずれかに該当するIssueはスキップする:

| 条件 | 判定方法 |
|------|---------|
| `planned` ラベルが付いている | `labels[].name` に `planned` が含まれる |
| コメントに実装計画が存在 | `comments[].body` に `## 実装計画` または `## Implementation Plan` が含まれる |

> ※ スキップ判定は Issue Scanner Agent が Phase 1-2 で実行する。

## 動的ワーカー数

| 未計画Issue数 | ワーカー数 |
|:---:|:---:|
| 0 | 処理終了（「全Issueが計画済みです」と報告） |
| 1〜5 | 1 |
| 6〜12 | 2 |
| 13+ | 3 |

> ※ ワーカー数の算出は Issue Scanner Agent が Phase 3 で算出し、`recommended_worker_count` として JSON に出力する。

## エラーハンドリング

| エラー | 対応 | リトライ |
|--------|------|---------|
| Issue Scanner Agent 失敗 | サブエージェント再起動 -> 失敗時リーダーが直接スキャン実行（フォールバック） | 1回 |
| issue-scan.json バリデーション失敗 | サブエージェント再起動 or リーダーフォールバック | 1回 |
| `OPENROUTER_API_KEY` 未設定 | `grok_status=skipped_auth` を記録して Codex 単独へ降格 | 0回 |
| Grok HTTP 402 / 429 | 1回だけ再試行。失敗時は `grok_status=timeout` | 1回 |
| Grok JSONパース失敗 | `grok_status=skipped_parse_error` を記録して Codex 単独へ降格 | 1回 |
| Grokタイムアウト（10分） | 部分出力があれば `partial_timeout` 保存。なければ `timeout` として Codex 単独へ降格 | 1回 |
| Codexタイムアウト/kill | 部分出力回収(tee) → プロンプト短縮 → reasoning引下げ → skip-git → スキップ（5段階、Bash timeout: 300000ms） | 段階的 |
| API 400 | プロンプト短縮 -> reasoning_effort=medium -> skip-git-repo-check -> 最終スキップ | 3回（段階的） |
| GitHubレートリミット | 全ワーカー一時停止 -> 60秒待機 -> 再開 | 自動 |
| コメント投稿失敗 | 1回リトライ -> 失敗時ローカル保存（/tmp/issue_plan_{number}.md） | 1回 |
| 認証エラー | 即リーダー報告 -> ユーザーフォールバック | 0回 |

### 段階的フォールバック
```
ワーカーリトライ（3回まで）
  | 失敗
別ワーカーに再割当
  | 失敗
リーダーが直接処理
  | 失敗
スキップ + ユーザーに報告
```

## アンチパターン

| やってはいけないこと | 理由 | 正しいやり方 |
|-------------------|------|-------------|
| Codexをメインスレッドで直接実行 | コンテキスト圧迫（出力が数百行） | ワーカーサブエージェント内で実行 |
| ラベル確認前にコメント投稿 | 二重投稿リスク | Step 4のスキップ判定を必ず先に実行 |
| コメント前にラベル追加 | 計画なしでラベルだけ付く | コメント投稿成功後にラベル追加 |
| 全Issueを1ワーカーに割当 | 並列化の意味なし | 動的ワーカー数テーブルに従う |
| レートリミット無視で連続実行 | GitHub APIブロック | 60秒待機を挟む |
| Codexタイムアウト時に無限リトライ | リソース浪費 | 段階的フォールバック（最大3回） |
| ワーカーがユーザーに直接質問 | ユーザー混乱 | リーダー経由でのみ（SendMessage） |
| Issue本文をそのままCodexに全文渡し | API 400エラー | 500文字に要約、2000文字超は冒頭500文字 |
| `gh issue list`にcomments/bodyを含める | JSON肥大化（数十KB超）->パース失敗 | 2パス方式: 軽量一覧->個別コメントチェック |
| shutdown_request直後にTeamDelete | アクティブメンバーがいるとTeamDelete失敗 | shutdown応答を確認してからTeamDelete |
| ワーカーのタスク取得順を指定しない | 両ワーカーが同じタスクを同時取得する競合リスク | 反対端方式 + ownerフィールドで防止 |
| ワーカーに停止条件を明示しない | 全タスク完了後もワーカーが自主的にタスクを作成・処理し続ける | プロンプトに「TaskListで未着手0件->終了」「タスク自作禁止」を明記 |
| リーダーがIssue一覧/コメントを直接取得する | リーダーのコンテキストにスキャン結果が蓄積し、パイプライン制御に不要なデータが残る | Issue Scanner Agent に委任し、JSON経由で結果を受け取る |
| TaskCreateのdescriptionにIssue bodyを埋め込む | Issue数x500文字がリーダーのコンテキストを圧迫 | number + title のみ含め、ワーカーが自分で `gh issue view` で取得 |
| レビュー結果をIssueコメントに含める | クリーンな計画だけ見せるべき | 内部管理のみ（レビュー痕跡はIssueに残さない） |
| レビュアー出力をコンテキストに全展開 | コンテキスト圧迫 | JSON形式で各1500トークン以内に制限 |
| Reviewer C未完了のまま統合に進む | レビュー品質低下（3/3->2/3） | 全レビュアー完了を待ってから統合。Bash timeout: 240000を必ず指定 |
| Reviewer Cタイムアウト即フォールバック | リトライすれば成功する可能性を放棄 | 短縮プロンプトで1回リトライしてからフォールバック判定 |
| 計画修正->再レビューのループ | タイムアウトリスク | 指摘を1回反映して投稿するだけ（再レビューしない） |
| **timeout未指定で調査コマンドを実行** | Bash toolデフォルト120秒で Scout / Main Analysis / Grok が途中終了する | Codex Scout=240000ms、Codex Main Analysis=300000ms、Grok=600000ms を明示指定する |
| **stdout依存（teeなし）でcodex exec実行** | kill/異常終了時に出力が全消失し、フォールバック判定すらできない | 全codex execに `2>&1 \| tee /tmp/codex_output_{number}.txt` を付加 |
| **kill後にCodexの従来フォールバック(プロンプト短縮等)を期待** | killは正常終了ではないため従来のフォールバック分岐に入らない | teeで永続化した部分出力を先に確認し、十分なら採用。不十分な場合のみ短縮版で再実行 |
| **Grok のファイルパスや行番号をそのまま採用** | Grok は補助分析であり、コード実在性を保証しない | ファイルパス・行番号・型・API可否は必ず Codex / Grep / Read で再確認する |
| **Xポスト由来知見を未検証のまま本文へ入れる** | 一次情報でないため、誤誘導や過剰設計の原因になる | `verification_required=true` を維持し、裏取りできた内容だけ計画へ反映する |
| **Grok の timeout を 5分未満に下げる** | 長文Issueや外部知見の整理が途中で切れやすい | Grok は既定で `600000ms` を使う |

## 検証方法

1. **ドライラン**: 小規模リポジトリ（2-3 open issues）で実行し、計画品質を確認
2. **コメント確認**: 投稿された実装計画がテンプレート通りか、Before/Afterコード付きか確認
3. **ラベル確認**: `planned` ラベルが正しく付与されているか確認
4. **再実行テスト**: 同じリポジトリで再実行し、計画済みIssueがスキップされるか確認
5. **エラーテスト**: 存在しないリポジトリ名を渡してエラーハンドリングを確認
6. **issue-scan.json検証**: スキャン結果JSONの全フィールドが正しく出力されているか確認
7. **フォールバック検証**: Issue Scanner Agent 失敗時にリーダーが旧 Step 2-3 を直接実行できるか確認
8. **ラベル適用検証**: 計画投稿後に `planned` ラベルが確実に付与されているか確認（教訓#6）
9. **Grokメタ検証**: `grok_status`, `grok_timeout_ms`, `grok_signals` が Issueコメントの meta に残るか確認
10. **信頼境界検証**: `verification_required=true` の Grok知見が最終計画本文に混入していないか確認

## ドライラン実績（2026-03-03）

**対象**: Robbits-CO-LTD/digital-management-consulting-app（19 open issues）

| 項目 | 結果 |
|------|------|
| 対象Issue | 10件 |
| スキップ | 9件（計画コメント済） |
| ワーカー数 | 2名（planner-1, planner-2） |
| 成功 | 10件（100%） |
| 失敗 | 0件 |
| P0検出 | 1件（#1072 サブスクキャンセル中に分析不可） |

**発見された改善点（適用済み）:**
1. `gh issue list --json` にcomments含めるとJSON肥大化 -> **2パス方式**に変更
2. ワーカー間タスク競合 -> **反対端方式 + ownerフィールド**で防止
3. shutdown_request直後のTeamDelete失敗 -> **応答確認フロー**追加
4. 完了レポートに優先度順テーブルがなかった -> **P0-P3ソート+工数合計**追加
5. ワーカーがスキップ対象Issueまで自主処理 -> **停止条件+タスク自作禁止**を明記
