# ウサコン チェックリスト — コードチェック

## 概要

Google スプレッドシートのチェック項目に対して、ソースコードの静的検査（コードチェック）のみを実行するスキル。
ワーカー（サブエージェント）にコード読み取りを委任し、OK/NG/UNSUREを判定する。

**参照先:**
- **親スキル**: [../checklist.md](../checklist.md) — 環境準備（Phase 1）、完了報告（Phase 3）、GitHub Issue作成、共通定義
- **Google Sheets操作**: `google-sheets-mcp` — セルナビゲーション・値読み取り・編集

**このスキルが行わないこと:**
- E2Eテスト（ブラウザでのUI操作テスト）
- G列（担当者）の操作
- usaconアプリへのブラウザアクセス

## アーキテクチャ: Taskツールによるサブエージェント委任

```
┌─────────────────────────────────────────────────┐
│              マネージャー（自分）                  │
│  役割:                                           │
│    - スプレッドシート読み取り・結果記録            │
│    - ワーカーへのコードチェック委任・結果集約       │
│    - GitHub Issue 作成                           │
│  ツール: playwright-cli(Bash経由), Bash(gh CLI)  │
└──────────────────────┬──────────────────────────┘
                       │ 1行ずつ逐次委任
                       │ （Taskツールでサブエージェント起動）
                       ▼
                 ┌───────────┐
                 │  checker  │  ← 行ごとに1つ起動
                 │  Row N    │
                 └───────────┘
                 役割: ソースコードを読み取り、
                       チェック項目に対してOK/NG/UNSUREを判定
                 ツール: Read, Grep, Glob, Bash（コード読み取り + 環境確認）
```

**設計方針**:
- **マネージャーがスプレッドシート操作を独占**: playwright-cliはメインセッション固有のため委任不可
- **ワーカーはコード読み取り専用**: ファイル書き込みなし → 所有権マップ不要
- **逐次委任**: マネージャーが1行読み取り → ワーカーに委任 → 結果を受けて記録、を繰り返す
- **Agent Teamsツール（TeamCreate/SendMessage等）は使用しない**: Taskツールによるサブエージェント起動のみ

| NGの種類 | ラベル | 例 |

**重要: UNSURE の場合は「関連コード断片」を必ず含める。** マネージャーがCodexにセカンドオピニオンを求める際、このコード断片をそのままプロンプトに埋め込むことで、Codexのファイル探索を不要にする。

**重要: エビデンスのない判定は不可。** 概要レベル（ファイル構造の把握）だけで「OK」と判定してはならない。必ず詳細レベル（具体的なコード行の引用、環境変数の実在確認）まで検証し、エビデンスを提示すること。環境変数が関わる項目では `vercel env ls` や `.env` ファイルで実際の設定を確認すること。

## 判定基準
- **OK**: コードが要件を満たしており、問題なし
- **NG**: 明確な不備・バグ・未実装がある
- **UNSURE**: 判定に迷う場合（ビジネスロジックの妥当性が不明、複数の解釈が可能、外部依存で確認不可など）
  → マネージャーがCodexでセカンドオピニオンを取得して最終判定する

## 検証レベルの分離（概要レベル確認 vs 詳細レベル検証）

> **「概要レベル確認」だけではチェック済みとしない。「詳細レベル検証」まで完了して初めて✅とする。**
> （教訓: 決済フローのコードチェックで概要分析のみで「確認済み」としたが、環境変数や具体的なコード行の検証を省略していた）

### レベル定義

| レベル | 内容 | 単独でのチェック完了 |
|--------|------|:---:|
| **概要レベル確認** | ファイル構造の把握、関数・クラスの存在確認、フロー全体の理解 | ❌ 不可 |
| **詳細レベル検証** | 具体的なコード行の引用、定数・マッピング値の正確性確認、環境変数の実在確認 | ✅ 必須 |

### 詳細レベル検証の必須チェック項目

1. **環境変数の検証**（環境変数が関わるチェック項目の場合）:
   - `vercel env ls` で本番・プレビュー環境に変数が設定されているか確認する
   - `.env` / `.env.local` / `.env.example` でローカル設定を確認する
   - コード内で `process.env.VARIABLE_NAME` として参照されている箇所を特定し、変数名の一致を検証する
   - **エビデンス例**: `vercel env ls` の出力結果、`.env.example` の該当行

2. **コード行レベルの引用検証**:
   - 「実装されている」と判定する場合、該当コードの**ファイルパス:行番号**を明示する
   - 定数・マッピング・設定値は具体的な値を引用して正確性を確認する
   - **エビデンス例**: `api/routes/payment.js:42` — `const PRICE_IDS = { basic: 'price_xxx', ... }`

3. **エビデンスの提示義務**:
   - ワーカーの結果報告に**必ずエビデンス（ファイルパス:行番号、コマンド実行結果）を含める**
   - 「問題なし」と判定する場合でも、確認した具体的な根拠を提示する
   - エビデンスのない「確認済み」判定は不可

## Codexによるセカンドオピニオン

ワーカーが UNSURE を返した場合や、マネージャー自身が判定に迷う場合に、Codex CLI（OpenAI）でセカンドオピニオンを取得する。**ユーザーに確認を求めず、自律的にCodexを呼び出して判定を継続する。**

**設計方針: MCP使用の最小化**

Codexにプロジェクトディレクトリを渡すと、ファイル探索でMCPツールを大量に消費する。これを防ぐため、**ワーカーが既に分析済みの情報（対象ファイルパス・コード断片・分析結果）をプロンプトに直接埋め込み、Codexはファイルアクセスなしで判定のみ行う**構成にする。

```
通常のCodex呼び出し:
  Codex → ファイル Read × N回 → Grep × M回 → 判定  ← コスト大

最小化後:
  ワーカー分析結果をプロンプトに埋め込み
  Codex → 判定のみ  ← ファイル探索実質ゼロ
```

**発動条件**:
- ワーカーの判定が `UNSURE`
- ワーカーが NG と判定したが、マネージャーが疑問を感じる場合
- コードの意図やビジネスロジックの妥当性が不明な場合
- 複雑なロジック（Stripe連携、Supabase RLS、認証フロー等）で確証が持てない場合

**実行方法**（codexスキル準拠: Taskツール Bashサブエージェント経由）:

マネージャーは、ワーカーの報告から**対象ファイルパスとコード断片**を抽出し、プロンプトに直接埋め込む。

**ワーカー返却フィールド → Codexプレースホルダー対応表**:
| ワーカー返却フィールド | Codexプレースホルダー |
|----------------------|---------------------|
| 判定 | `{ワーカーの判定}` |
| 対象ファイル | `{ワーカーが確認したファイルパス}` |
| 確認内容 | `{ワーカーの確認内容要約}` |
| 問題点 | `{ワーカーが指摘した問題点}` |
| 関連コード断片 | `{ファイルパス1}` + `{コード断片1}` ... |

```

    ファイル: {ファイルパス2}
    {コード断片2}

    上記を基に OK/NG を判定し、理由を簡潔に述べてください。確認や質問は不要です。
    ---ここまで---

    ステップ2: Codex実行（Bashツール、timeout: 120000）
    codex exec --full-auto --sandbox read-only "$(cat /tmp/codex-prompt.txt)"

    ステップ3: 結果要約
    以下の形式で結果を要約してください：
    1. Codexの最終判定（OK / NG）
    2. 判定理由（1〜2文）
    3. 問題点（NGの場合のみ、1文）
    4. 推奨修正（NGの場合のみ、1文）
```

**重要**: Bashツールの `timeout` パラメータを `120000`（120秒=2分）に設定すること。

**`--sandbox read-only`** を使用する。プロンプトに必要な情報をすべて埋め込んでいるため、Codexがファイル探索する必要はない。これにより:
- ファイル探索: 実質ゼロ（プロンプト内の情報で判定完結）
- タイムアウト: 300秒→120秒に短縮（探索不要のため高速）
- API消費: 最小限（プロンプト内コンテキストのみ処理）

**マネージャーの準備作業**:
Codex呼び出し前に、ワーカーの報告からプロンプトに埋め込む情報を整理する:
1. ワーカーが報告した「対象ファイル」のパス一覧
2. ワーカーの「確認内容」と「問題点」の要約
3. 判定に関わる**コード断片**（ワーカーの報告に含まれる場合はそのまま使用、なければマネージャーが Read で取得して埋め込む）

**最終判定ルール**:
| ワーカー判定 | Codex判定 | 最終判定 |
|-------------|----------|---------|
| OK | — | OK（Codex不要） |
| NG | — | NG（Codex不要、明確な不備） |
| UNSURE | OK | OK |
| UNSURE | NG | NG |
| NG（マネージャーが疑問） | OK | OK（Codex優先） |
| NG（マネージャーが疑問） | NG | NG（一致、確定） |

**注意**:
- Codex呼び出しはAPI利用料金が発生するため、明確にOK/NGが判定できる場合は呼び出さない
- Codexが返した判定もあくまで参考。最終的にはマネージャーが責任を持って判定する
- Codex実行は必ずTaskツール（Bashサブエージェント）経由で行い、メインコンテキストを保護する
- **Codexにコードを探索させない** — 必要な情報はすべてプロンプトに埋め込む

## エラーリカバリ

**基本方針**: チェック失敗は即座にD列・E列に記録し、次の項目に進む。1つの失敗で全体を停止しない。

| エラーケース | リカバリ手順 |
|-------------|-------------|
| **ワーカー（Task）がタイムアウト** | 同じ行を1回リトライ → 再度失敗なら UNSURE 扱いでCodexエスカレーション |
| **Codex CLI がエラー** | Codexをスキップし、ワーカーの分析結果のみで判定（マネージャー独自判定）。E列に「Codex判定不可」と補記 |
| **`gh issue create` が失敗** | F列に「Issue未作成」と記録し続行。完了報告でまとめて未作成Issueを報告 |
| **スプレッドシート書き込み失敗** | ページリロード（F5）後にリトライ。2回失敗なら当該行をスキップして報告 |
| **ブラウザセッション切れ（Playwright MCPアイドルタイムアウト）** | 下記「ブラウザセッション維持と復旧」を参照 |

### ブラウザセッション維持と復旧

サブエージェント（Taskツール）でコードチェックを並列実行した場合、完了待ちの間にPlaywright MCPのブラウザプロセスがアイドルタイムアウトで終了することがある。

**原因**: Playwright MCPプラグインは一定時間操作がないとブラウザプロセスを終了する。コードチェックのサブエージェントは完了まで数分かかるため、その間スプレッドシートへの操作が途絶えてタイムアウトに達する。

#### 対策1: コードチェック中のブラウザキープアライブ

サブエージェント完了待ちの間、定期的にスプレッドシートタブで軽い操作を行いセッションを維持する:
- `browser_take_screenshot` でスプレッドシートのスクリーンショットを撮る
- `browser_snapshot` でアクセシビリティスナップショットを取得する
- 目安: **60秒に1回**程度の操作で十分

```
サブエージェント起動
  ↓
待機中に browser_take_screenshot（60秒間隔）
  ↓
サブエージェント完了通知
  ↓
結果記録へ
```

#### 対策2: ブラウザが閉じてしまった場合の復旧手順

1. `browser_navigate` でスプレッドシートURLを再度開く
   - URL: `https://docs.google.com/spreadsheets/d/1h1MSOUuH2hx0-Q-FCFjp6ylisY9yc4Y6EAocToFXUFg/edit`
2. 「アプリ」シートタブが選択されていることを確認（されていなければクリック）
3. Name Boxで対象行にジャンプ（`A{行番号}`）— google-sheets-mcp セクション3参照
4. スクリーンショットで現在の状態を確認し、作業を再開

#### 対策3: 推奨ワークフロー（一括記録方式）

コードチェック結果をまとめて記録する場合は、**全サブエージェント完了後にブラウザを開き直して一括記録する**方が安定する:

1. サブエージェントを起動し、全チェック結果が揃うまで待つ（ブラウザ操作なし）
2. 全結果が揃ったら `browser_navigate` でスプレッドシートを開く
3. 結果をまとめてスプレッドシートに記録する

この方式であれば、ブラウザセッション切れを気にする必要がない。ただし、逐次記録方式（1行チェック→即記録）と比べて、途中で失敗した場合に未記録の結果が多くなるリスクがある点に注意。
