# ウサコン チェックリスト — E2Eテスト

## 概要

Google スプレッドシートのチェック項目に対して、playwright-cli によるブラウザ操作テスト（E2Eテスト）のみを実行するスキル。
アプリのUI操作ナレッジに基づいて自律的にテストを実行し、結果をD列に記録、G列に担当者を設定する。

**参照先:**
- **親スキル**: [../checklist.md](../checklist.md) — 環境準備（Phase 1ステップ1-3）、完了報告（Phase 3）、GitHub Issue作成、共通定義
- **Google Sheets操作**: `google-sheets-mcp` — セルナビゲーション・値読み取り・編集
- **UI操作ナレッジ**: [app-knowledge.md](app-knowledge.md) — アプリUI構造・ページ別操作ガイド・共通操作パターン

**このスキルが行わないこと:**
- コードチェック（サブエージェント委任によるソースコード検査）
- A列（コードチェック✅）の操作
- Codexエスカレーション

## Phase 1: 環境準備

親スキル（[../checklist.md](../checklist.md)）の共通Phase 1（ステップ1-3）+ E2E固有の準備（ステップ4-7）を行う。
**前提**: 親スキル経由（mode=both）で呼ばれた場合、ステップ1-3は既に実行済みのため再実行しない。ステップ4-7のみ実行する。

```
=== 共通準備（ステップ1-3）===
親スキルのPhase 1を参照。単独実行時のみ実行。

=== E2E固有の準備（ステップ4-7）===
4. usaconプレビューサイト用の新タブを作成（E2Eテスト用）
   playwright-cli tab-new https://preview.usacon-ai.com
5. ログイン操作を実行（app-knowledge.md「共通操作パターン > ログイン操作」参照）
   ※ プラン別テストが必要な場合は上記テストアカウント表から適切なアカウントを使用
6. ダッシュボード表示を確認
7. tab-list でタブのインデックスを記録（スプレッドシート: 0, usacon: 1 等）
   ※ 以降タブ切替は playwright-cli tab-select <index> で行う
```

## Phase 2: E2Eテストループ（マネージャー主導）

```
ループ（対象行ごと）:

  === ① スプレッドシートから行データ読み取り ===
  a. スプレッドシートタブに切替（playwright-cli tab-select {スプレッドシートIndex}）
  b. Name Boxで対象行にジャンプ（A{行番号}）— google-sheets-mcp セクション3参照
  c. スクリーンショットでB列（内容）・C列（テスト方法）・D列（結果）・F列（備考）・G列（担当者）を読み取り

  === ② スキップ判定 ===
  d. 以下の場合はスキップ:
     - 水色背景のセクションヘッダー行 → スキップ
     - G列に値あり（既にテスト済み）→ スキップ
     - D列に「NG」が入っている行 → スキップ（既にNG判定済み）
     - C列が明確にコードチェック専用（下記キーワード参照）→ スキップ
     ※ A列の✅はE2Eテストのスキップ条件ではない（コードチェックの状態を示すものであり、E2Eテストとは独立）

  === ③ E2Eテスト要否判定 ===
  e. E2Eテストの要否を判定:
     - **E2Eテスト必要**: C列に以下のキーワードを含む場合
       → 「ブラウザで確認」「画面遷移」「表示確認」「クリック」「操作」「入力」「ドロップダウン」「ボタン」「ログイン」
     - **コードチェックのみ（E2E不要）**: C列に以下のキーワードを含む場合
       → 「コードレビュー」「実装確認」「ソースコード確認」「コード確認」「設計確認」
     - **迷う場合**: C列の内容を総合的に判断。UI操作を伴う検証はE2E、静的なコード検査はコードチェックのみ
  f. E2Eテスト不要と判定された行 → スキップ（コードチェックスキルの担当）

  === ④ E2Eテスト実行 ===
  g. usaconタブに切替（playwright-cli tab-select {usaconIndex}）
  h. C列の手順 + app-knowledge.md のUI操作ナレッジに基づきブラウザテスト実行（playwright-cliスキル参照）
  i. スクリーンショットで結果確認（playwright-cli screenshot）

  === ⑤ 結果記録 ===
  j. スプレッドシートタブに戻り:
     - **PASS の場合**:
       1. D列: 記入しない（空のままにする。D列はNG時のみ使用）
       2. G列に担当者「竹下」を設定（google-sheets-mcp セクション8 方法B — JavaScript execCommand方式を使うこと）
       ※ **複数行のG列を連続設定する場合**: Name Boxは最初の1行目のみ使用し、以降はEnter確定の自動移動で次行に進む（google-sheets-mcp セクション12参照）。**Enter後にName Boxで再ジャンプするとA列に誤入力するリスクあり**
     - **FAIL の場合**:
       1. D列に「NG」、E列に原因を記入
       2. GitHub Issue を作成（親スキル「不備時のGitHub Issue作成」参照）
       3. F列にIssue番号を追記
       4. G列に担当者「竹下」を設定

  k. 次の行へ
```

## テスト判定基準

| テストタイプ | 判定方法 | 成功条件 |
|-------------|---------|---------|
| ページ表示確認 | URLが正しい + ページ要素が表示 | 期待URLに遷移 + 主要コンテンツ表示 |
| UI操作確認 | 操作後の状態変化 | ボタンクリック後に期待する変化あり |
| フィルター確認 | フィルター操作後のリスト変化 | 絞り込み結果が反映される |
| API確認 | ネットワークリクエスト | `playwright-cli network` で200確認 |
| AI応答確認 | レスポンス受信 | 何らかのテキストが表示（品質は人間判断） |
| DB確認 | ページ上のデータ表示 | 登録・更新データがUIに反映 |
| エラーハンドリング | エラー状態の表示 | エラーメッセージ or フォールバック |
| Stripe連携 | Checkout遷移・Webhook | セッションID確認・イベント配信確認 |

## 注意事項

### Google Sheets 操作時の注意
- 座標ベースの操作は避け、google-sheets-mcp スキルの refベース操作パターンに従うこと
- すべての操作は `playwright-cli` コマンド（Bash経由）で実行。操作後は必ず `snapshot` を再実行してrefを更新
- 編集モード中に別セルをクリックすると意図しない場所に入力される。必ずEnterで確定してから次の操作へ
- シートタブが「アプリ」になっていることを確認（「管理画面」タブと間違えない）

### テスト実行時の注意
- 新規登録後は `/login` にリダイレクトされる（`/email-confirmation` ではない）。確認メールは迷惑メールフォルダに届く場合がある
- メール認証のリンククリックは人間が行う必要がある（E2Eテスト中断→人が確認→続行）
- AI応答系テスト（分析実行・レポート生成・チャット）は応答に30秒〜2分かかるため、wait で十分待機
- Stripe決済系テストはpreview環境のテストモードで実行（テストカード: 4242 4242 4242 4242）
- ログインセッションが切れた場合は再ログインしてからテスト続行
- 企業選択が必要なテストでは、ドロップダウンで適切な企業が選択されているか確認

## エラーリカバリ

**基本方針**: テスト失敗は即座にD列・E列に記録し、次の項目に進む。1つの失敗で全体を停止しない。

| エラーケース | リカバリ手順 |
|-------------|-------------|
| **プレビューサイトのログイン切れ** | `tab-select {usaconIndex}` → 再ログイン → テスト続行 |
| **連続失敗3件以上** | ログインセッション切れ・サーバーエラーを疑い、`tab-select {usaconIndex}` → `screenshot` でダッシュボード表示を確認してから続行 |
| **`gh issue create` が失敗** | F列に「Issue未作成」と記録し続行。完了報告でまとめて未作成Issueを報告 |
| **スプレッドシート書き込み失敗** | ページリロード（F5）後にリトライ。2回失敗なら当該行をスキップして報告 |

### Stripe決済テストの制限
- 実際の決済完了・Webhook受信確認はE2Eテストの範囲を超える場合がある
- Checkout画面への遷移確認、URL・セッションIDの確認までをE2Eテストの範囲とする
- Webhook受信はStripeダッシュボードでの確認が必要
