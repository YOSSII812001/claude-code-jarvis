# 計画レビューモード（`/sub-review --plan`）

> Issue #1711レビュー（2026-04-04）で有効性を実証。
> Codex×2（外部LLM独立検証）+ サブエージェント×2（漏れ/いじわる）で計画の品質を検証する。
> **差分モードとの最大の違い: エージェントが実コードを探索する**（diff埋め込みではない）。

## Plan Step 1: 計画の正規化（Canonical Input）

レビュー対象を一意に確定してから全Agentに渡す。曖昧な場合はレビュー開始前にユーザーに確認する。

**入力の正規化手順:**

1. 引数がGitHub Issue URLの場合:
   ```bash
   gh issue view <番号> --repo <owner/repo> --json title,body,comments
   ```
2. Issueコメントに「実装計画」がある場合は **最新の `planned` ラベル付き計画コメント** を正本とする
3. 複数の計画コメントがある場合は **最新のもの** を採用（古い計画は無視）
4. 計画テキストが直接渡された場合はそのまま使用する

**正規化後の入力束（全Agentに共通で渡す）:**
- Issue title + body（要件の原文）
- 採用した計画コメント（正本）
- E2E受け入れ条件（計画に含まれる場合）

## Plan Step 2: 影響ファイルの特定 + CLAUDE.md確認

計画から変更対象ファイルを抽出し、**補完探索** で関連ファイルも特定する:

```
1. 計画に明示されたファイル → 存在確認（ls）
2. シンボル参照（Grep）→ 呼び出し元・型定義
3. テストファイル（Glob）→ 関連テスト
4. DB migration/schema → スキーマ整合確認用
5. CLAUDE.md → プロジェクト規約の確認（Agent 3/4のプロンプトに含める）
```

計画にファイルパスが記載されていない場合は「計画不備」として報告に含める。

## Plan Step 3: 4つの専門 Task を並列起動

以下の4つを **1つのメッセージで並列起動** する（Agent tool × 4）。
**計画モードではエージェントが実コードを読む**（差分埋め込み方式ではない）。

### Plan Task 1: Codex — 構造・依存・インターフェース互換性

**焦点**: モジュール配置、公開インターフェース互換性、依存方向、データフロー

```
Agent(
  description="Codex: コード構造レビュー",
  prompt="""
  あなたはCodex CLIを使って、実装計画を「構造・依存・インターフェース互換性」の視点からレビューする。

  ## 正規化済み入力
  ### Issue要件
  {Issue title + body}
  ### 実装計画（正本）
  {採用した計画コメント}

  ## 専門レビュー観点（他Agentとの責務分離: DB/migration/ロールアウトはTask 2が担当）
  1. モジュール構造・ファイル配置は適切か
  2. 関数シグネチャの互換性（呼び出し元への影響）
  3. データフローに構造的問題はないか（非破壊コピー、参照共有等）
  4. 既存コードとの結合度（import先の変更影響）
  5. 型整合性（フロントエンド↔バックエンドの型定義）
  6. module.exportsの変更がテスト以外の外部呼び出しに影響しないか

  ## タスク
  1. 計画に記載された変更対象ファイルを全て読む
  2. 呼び出し元、型定義、関連エクスポートも読む
  3. Codex CLIを呼び出して上記観点でレビューを依頼
  4. 計画のBefore/Afterと実コードを比較して乖離を検出
  5. 結果をCritical/Warning/Info形式でまとめる

  レビュー結果は省略せず、最終回答として必ず全文を出力してください。
  """
)
```

### Plan Task 2: Codex — 実現可能性・DB・ロールアウト整合

**焦点**: DBスキーマ整合、参照先の実在確認、数値見積もり、デプロイ/ロールアウト整合

```
Agent(
  description="Codex: 実現可能性レビュー",
  prompt="""
  あなたはCodex CLIを使って、実装計画を「実現可能性・DB・ロールアウト整合」の視点からレビューする。

  ## 正規化済み入力
  ### Issue要件
  {Issue title + body}
  ### 実装計画（正本）
  {採用した計画コメント}

  ## 専門レビュー観点（他Agentとの責務分離: 構造・依存はTask 1が担当）
  1. DBスキーマとの整合（カラム名・型の一致、migration順序）
  2. 計画が参照する既存関数・定数・型が実際に存在するか

  4. 「そのまま実装して動くか」を厳密に検証
  5. 結果をCritical/Warning/Info形式でまとめる

  レビュー結果は省略せず、最終回答として必ず全文を出力してください。
  """
)
```

### Plan Task 3: 漏れチェック

```
Agent(
  description="漏れチェックレビュー",
  prompt="""
  実装計画を「漏れ・抜けがないか」の視点でレビューする。

  ## 正規化済み入力
  ### Issue要件
  {Issue title + body}
  ### 実装計画（正本）
  {採用した計画コメント}
  ### プロジェクト規約
  {CLAUDE.mdの要点}

  ## タスク
  1. 計画に記載された全ファイル + Step 2で特定した関連ファイルを読む
  2. 以下の観点で漏れをチェック:
     a. フィールド/パラメータの網羅性（計画の対象 vs Issue要件 vs 実コード）
     b. 呼び出し元の漏れ（変更関数を呼ぶ箇所が他にないか）
     c. テスト計画の漏れ（検証すべきだが記載されていないシナリオ）
     d. エラーハンドリングの漏れ（未対処のエラーパターン）
     e. 並行実行・競合状態の考慮漏れ
     f. ロールバック時のデータ整合性
     g. ログ/監視の不足
     h. デプロイ手順・migration順序の漏れ
     i. ドキュメント更新の漏れ
     j. CLAUDE.md規約との整合性
  3. 各項目「漏れあり/漏れなし」を明記し、具体的な内容と対策を記載

  レビュー結果は省略せず、最終回答として必ず全文を出力してください。
  """
)
```

### Plan Task 4: いじわるチェック

```
Agent(
  description="いじわるチェックレビュー",
  prompt="""
  実装計画を「いじわるテスト」の視点でレビューする。
  意地悪な入力、想定外の状況、壊れやすいポイントを徹底的に洗い出す。
  **到達可能な攻撃面のみを検討すること**（理論上のみのリスクは現実性「極低」と明記）。

  ## 正規化済み入力
  ### Issue要件
  {Issue title + body}
  ### 実装計画（正本）
  {採用した計画コメント}

  ## タスク
  1. 計画に記載された全ファイル + 関連する型定義 + DBスキーマを読む
  2. 以下の観点でいじわるチェック:
     a. 悪意のある入力データ（到達可能なXSS、超長文、負数、型不正）
     b. 壊れやすいマッピング/変換（ケース不一致、フォーマット揺れ、空配列）
     c. タイミング/状態の問題（並行更新、abort、競合状態）
     d. 型変換の罠（String(null)="null"、String({})="[object Object]"等）
     e. 環境依存の罠（TZ、Node.jsバージョン、DB返却型）
     f. マージ/上書きの罠（順序依存、キー衝突）
     g. 段階デプロイ時の旧新互換（新コード+旧DB、旧コード+新DB）
     h. 権限/テナント境界（org_id漏れ、RLS迂回）
     i. セキュリティ（ログへの個人情報漏洩、認証バイパス）
  3. 各シナリオの深刻度（Critical/Warning/Info）と**現実性（高/中/低/極低）**を評価

  レビュー結果は省略せず、最終回答として必ず全文を出力してください。
  """
)
```

## Plan Step 4: 結果の統合

4つの Task の結果を受け取った後：

1. **深刻度で分類**: Critical → Warning → Info
2. **クロスリファレンス（Evidence Strength）**: 何視点から指摘されたかを明記（1-4）
3. **通過条件の判定**:
   - Critical = 0件 → **通過**（実装開始OK）
   - Critical ≥ 1件 → **不通過**（計画修正後に `/sub-review --plan` を再実行し、Critical = 0 を確認するまで閉じない）
   - Evidence Strength ≥ 2 の Warning → 実装時に必ず対処（推奨アクションに含める）
4. **推奨アクションリスト**: Critical → Evidence Strength ≥ 2 の Warning → 残りの Warning の順
